# P5.5 — Cutover (homepage move, indexability, sitemap, structured data, redirect, deploy) — Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Land the public-launch atomic cutover. P5.5 is the *last* phase of the P5 series and ships a single deployable diff that flips CentralGauge from "private internal review" to "indexable public site". After this plan merges + deploys:

1. **Leaderboard becomes the homepage** (`/`). The placeholder `<h1>CentralGauge</h1><p>API-only build (P1). UI ships in P5.</p>` at `site/src/routes/+page.svelte` is replaced by the leaderboard markup. The standalone `/leaderboard` URL is retired (with a 30-day 302 redirect; architect I4: temporary, not 301; see Mini-phase B).
2. **Indexability unlocks** — `<meta name="robots" content="noindex">` is removed from `site/src/app.html`, `static/robots.txt` is published with `Allow: /` + sitemap pointer, and the build emits `sitemap.xml` listing every public route (architect I9: artifact lives in `.svelte-kit/cloudflare/sitemap.xml`, not committed; served via the SvelteKit `[assets]` binding).
3. **SEO baseline** — per-page `<link rel="canonical">` populated by the layout, JSON-LD `WebSite` + `Organization` structured data emitted in `<svelte:head>`, OG tags audited.
4. **SSE route map updated** — `eventToRoutes()` and the `useEventSource(['/leaderboard'])` callsites in 3 files swing to `/`. Existing SSE clients reconnect naturally on next page load (the worker reload during deploy evicts active streams; the client retry path opens fresh against the new path).
5. **All consumers re-pointed** — Lighthouse URLs, Playwright specs, Nav active-route, internal `<a href>`s, doc references.
6. **Single atomic deploy** — one `wrangler deploy` flips production. The branch is reviewed at canary (`/_canary/<sha>/`) before merge.

**Architecture:** Three structural moves (route file relocation, app.html meta deletion, layout `<svelte:head>` additions); one new static asset directory (`site/static/`) with a generator script (`scripts/build-sitemap.ts`); one new endpoint (`/leaderboard/+server.ts` 302 redirect, sunset 2026-05-30); existing-route patches (`useEventSource(['/leaderboard'])` → `useEventSource(['/'])` per architect I8: 8 occurrences in 3 files); 14 spec/doc updates. No new components, no new types, no new flags. The flag matrix carried forward from P5.4 is unchanged.

> **Design rationale: cutover is one atomic diff, not staged.** Spec §11.3 P5.5 row says "Replace placeholder homepage with leaderboard; publish sitemap + robots." Single phase. We could split into "first publish sitemap+robots silently while still noindex'd, then a week later flip /leaderboard → /" but that doubles deploy risk for zero observable safety: external bookmarks to `/leaderboard` either redirect (if we ship the 30-day 302) or 404. There is no graceful "soft launch" mode we get for free by staging. Land it as one PR, review at canary, deploy once.

> **Design rationale: 302 redirect with a 30-day sunset for `/leaderboard` (architect I4).** External bookmarks may exist (CI dashboards, Slack history, internal docs we haven't audited). A 302 (Found, temporary) redirect from `/leaderboard` → `/` is one cheap SvelteKit `+server.ts` per the Kit endpoints API. We preserve query strings (`?tier=verified` etc.) so bookmarked filtered views still land correctly. The redirect is set to expire on **2026-05-30** (30 days post-cutover) — at which point the route file deletes and `/leaderboard` returns the SvelteKit 404 (`+error.svelte`). The expiry is enforced by a CI test that fails 14 days BEFORE 2026-05-30 (architect C5 reminder-window pattern; not a runtime `Date.now() > expiry` check, which would silently flip behavior on an arbitrary day — bad). 302 is the right status code for a temporary redirect (the route is going away — we don't want browsers to cache the redirect indefinitely after the file is deleted). The sunset window is reversible: if we discover external bookmarks rely on the redirect, extend the expiry by editing `SUNSET_ISO` in the test; the redirect file itself is unchanged.

> **Design rationale: sitemap is static-at-build, not dynamic-at-request.** Two options:
>
> - **Static**: `scripts/build-sitemap.ts` runs in `npm run build` and emits `.svelte-kit/cloudflare/sitemap.xml` from a hardcoded route list (architect I9: not committed). Deploys regenerate. Cacheable forever via the SvelteKit `[assets]` binding.
> - **Dynamic**: `src/routes/sitemap.xml/+server.ts` returns the list at request time; could include dynamic entries (e.g. one URL per indexed model slug from D1).
>
> Static wins on simplicity, cacheability, and zero D1 cost-per-crawl. We don't currently have model/run/task pages we *want* indexed individually — they exist for direct link-share but the leaderboard is the canonical entry point. If P6+ wants per-model deep links indexed, swap to dynamic via a follow-up plan; the sitemap consumer (Googlebot) handles either format identically. Static-now leaves that door open without forcing the decision today.

> **Design rationale: structured data is layout-level (`WebSite` + `Organization`), not per-page.** Spec §5.8 says "structured data via JSON-LD". We have two reasonable scopes:
>
> - **Layout-only**: one `WebSite` schema (with `potentialAction` sitelinks searchbox pointing at cmd-K) + one `Organization` schema. Both render on every page. Total: ~600 bytes JSON in `<head>`.
> - **Per-page**: layout-only PLUS each page declares its own `Article`/`Dataset`/`SoftwareApplication`/etc.
>
> Per-page lets richer rich-results show in Google search (e.g. cards for `/runs/<id>` could link directly to the transcript). But it requires deciding the right schema per route (Article? Dataset? Report?), populating each loader with extra fields, and getting the JSON validated. Per-page is **deferred to P6** along with structured-data validation in CI. P5.5 ships the layout-only minimum so we have *something* and search engines see consistent canonical + organization signals from day one.

> **Design rationale: SSE clients survive cutover via natural reconnect.** Existing P5.4 clients hold open `EventSource('/api/v1/events/live?routes=%2Fleaderboard')` connections to the worker. At deploy time:
>
> 1. Cloudflare gracefully drains the old worker version's open streams (typically ~30 s tolerance window).
> 2. The new worker version receives traffic; its `eventToRoutes()` no longer maps any event to `/leaderboard` — it maps to `/`.
> 3. The drained client's `EventSource.onerror` fires; `useEventSource`'s 1s/3s/10s exponential backoff reconnects.
> 4. **But** the client still sends `routes=%2Fleaderboard` because the page hasn't navigated. Server-side filter rejects every event (no event-routes overlap). The status pill shows "live" but no invalidations fire.
> 5. The client only switches to `routes=%2F` when the user reloads or navigates.
>
> This is the *correct* failure mode: a stale tab held open across cutover doesn't get garbage events, and the next interaction repairs the connection. The alternative — server-side aliasing `/leaderboard` to `/` in `routePatternMatches` — is more code and only papers over a transitional symptom that resolves on first page interaction. We document this in `docs/site/operations.md` post-cutover-checks instead.

> **Design rationale: bookmarked filter URLs are preserved through the 302.** A user with `https://centralgauge.sshadows.workers.dev/leaderboard?tier=verified&sort=avg_score:desc` bookmarked must land on the equivalent `/` URL with the same filters. The `+server.ts` redirect captures `event.url.search` and concatenates `Location: '/' + url.search`. Tested in `site/tests/api/leaderboard-redirect.test.ts` — 4 cases (no params, single param, multiple params, encoded special chars).

> **Design rationale: cache invalidation is benign — old `cg-leaderboard` named-cache entries strand for ≤ 60 s.** P5.1 introduced the `cg-leaderboard` Cache API named cache for the `/api/v1/leaderboard?…` API responses. The Cache API key is the request URL; the URL doesn't change at cutover (the API endpoint is unaffected, only the SSR'd HTML page route moves). However, SvelteKit's adapter-cloudflare *also* caches the SSR'd HTML response in `caches.default` keyed by the request URL — so old `/leaderboard` HTML may still be served by some colos for ≤ s-maxage seconds (60 s per `+page.server.ts`'s `setHeaders('cache-control')`). This is acceptable: 60 s is below the time-to-meaningful-paint for most users, and the URL the cache was keyed on (`/leaderboard`) becomes a 302 redirect after cutover, so cache hits effectively serve the old leaderboard HTML while the URL bar redirects on the *next* navigation. We do NOT proactively purge `caches.default` at deploy time — Cloudflare offers no API to do so from the worker without a paid plan. Document this transient in `operations.md`.

> **Design rationale: prerender skip for `/` (architect R2).** The `prerender.entries` config in `svelte.config.js` lists `/about` (the only prerender candidate today). The new `/` route is dynamic (per-request leaderboard data, SSE-tagged) and MUST NOT be prerendered. SvelteKit's default for routes not in `prerender.entries` is "do not prerender" — the new `/` inherits this default. To make the intent explicit and stop a future routes-config sweep from accidentally prerendering it, add `export const prerender = false;` at the top of the new `+page.server.ts` (covered in B2; see Task B2 Step 1).

**Tech Stack:** Same as P5.4 (Svelte 5.55+, Kit 2.58+, `@cf-wasm/og`, `@noble/ed25519`). No new runtime deps. One new build-time dep (deferred behind a build script — `xmlbuilder2` is overkill for a hardcoded URL list; we hand-write the XML string). Sitemap generation lives at `scripts/build-sitemap.ts` and runs in the existing `npm run build` chain via a pre-build step. The `static/` directory is created from scratch (none today per `Glob('site/static/**')` returning zero results).

## Mini-phase A — Static assets + sitemap generator

Lays the groundwork for indexability. New `static/` directory with `robots.txt` + `sitemap.xml`, plus a build-time generator script. None of these change runtime behavior on its own — A1–A3 are pure additions. The cutover (B*) and noindex removal (C1) are what activate them.

### Task A-1: Create `$lib/shared/site.ts` — single source of truth for `SITE_ROOT`

**Files:**
- Create: `site/src/lib/shared/site.ts`
- Create: `site/src/lib/shared/site.test.ts`
- Modify: `site/wrangler.toml`

The plan emits the absolute base URL (`https://centralgauge.sshadows.workers.dev`) in three places: sitemap, JSON-LD schemas, canonical link. Hardcoding it in three modules makes the P7 custom-domain swap a multi-file edit. Centralize once.

> **Design rationale: env var with default, not D1.** The base URL is a config fact, not a data fact — it changes once per deploy environment, never per request. Reading from `process.env` at module-load time means: build-time scripts (`build-sitemap.ts`) pick it up via `npm run build`; runtime SSR picks it up from wrangler `[vars]`. Tests can override via `vi.stubEnv('SITE_BASE_URL', '...')` without touching code.

```ts
#!/usr/bin/env tsx
/**
 * Sitemap generator. Run at build time (`npm run build` pre-step). Emits
 * `.svelte-kit/cloudflare/sitemap.xml` from a hardcoded route list.
 *
 * Why static, not dynamic at request time: simpler, cacheable forever,
 * zero D1 cost-per-crawl, deterministic (test snapshot bites). When P6+
 * wants per-model deep links indexed, swap to dynamic via a follow-up
 * plan; consumers (Googlebot) handle either format identically.
 *
 * Why hardcoded routes (not D1-driven): the public route list is a
 * design decision, not a data fact. We choose to advertise the
 * leaderboard, the eight cross-cut surfaces, and About — but not
 * `/runs/<id>` (those are deep links, not landing pages users would
 * enter via search). Hardcoding makes that explicit and PR-reviewable.
 *
 * Sunset of /leaderboard: NOT included. Pre-cutover the route was
 * accessible at /leaderboard but noindex'd, so it never made it into
 * any prior sitemap. Post-cutover it's a 302 redirect — listing it
 * would tell crawlers to follow the redirect (wasteful) or worse,
 * index it alongside / (duplicate-content signal).
 */
import { mkdirSync, writeFileSync } from 'node:fs';
import { dirname, resolve } from 'node:path';
import { fileURLToPath } from 'node:url';

// SITE_BASE_URL is the single source of truth. Read at module-load time
// so build-time invocations (`npm run build`) and tests both pick up the
// same value. Default mirrors `wrangler.toml` `[vars] SITE_BASE_URL`.
// When P7 lands a custom domain (e.g. `centralgauge.dev`), set
// `SITE_BASE_URL=https://centralgauge.dev` in the build environment AND
// edit `wrangler.toml` — both paths re-target atomically.
export const BASE_URL =
  process.env.SITE_BASE_URL ?? 'https://centralgauge.sshadows.workers.dev';

// ALPHABETIZED + DEDUPLICATED. Tests assert sortedness.
export const SITEMAP_ROUTES: ReadonlyArray<string> = [
  '/',
  '/about',
  '/compare',
  '/families',
  '/limitations',
  '/models',
  '/runs',
  '/search',
  '/tasks',
];

export function buildSitemap(): string {
  const lines: string[] = [];
  lines.push('<?xml version="1.0" encoding="UTF-8"?>');
  lines.push('<urlset xmlns="http://www.sitemaps.org/schemas/sitemap/0.9">');
  for (const route of SITEMAP_ROUTES) {
    // Homepage emits with explicit trailing slash to match the canonical
    // `<link rel="canonical">` value (which is `${BASE_URL}/` for `/`).
    // Without this both the sitemap and the canonical disagree on whether
    // the homepage URL has a trailing slash, which Google flags as a
    // duplicate-canonical signal.
    const loc = route === '/' ? `${BASE_URL}/` : `${BASE_URL}${route}`;
    lines.push('  <url>');
    lines.push(`    <loc>${loc}</loc>`);
    lines.push('    <changefreq>daily</changefreq>');
    lines.push('  </url>');
  }
  lines.push('</urlset>');
  // Trailing newline — POSIX text-file convention; some XML linters flag
  // its absence.
  lines.push('');
  return lines.join('\n');
}

// Standard Node ESM entrypoint detection. `import.meta.main` is a Deno-ism
// (undefined in Node + tsx); using it here would silently fall through and
// always treat this file as a library import.
const isMain =
  process.argv[1] !== undefined &&
  fileURLToPath(import.meta.url) === resolve(process.argv[1]);

if (isMain) {
  // SvelteKit's adapter-cloudflare emits the worker bundle to
  // `.svelte-kit/cloudflare/`; static files copied into that bundle are
  // served by the [assets] binding. We write the sitemap there directly so
  // the artifact is rebuilt every CI run and never committed (see I9).
  const target = resolve(process.cwd(), '.svelte-kit/cloudflare/sitemap.xml');
  mkdirSync(dirname(target), { recursive: true });
  const xml = buildSitemap();
  writeFileSync(target, xml, 'utf8');
  console.log(`[sitemap] wrote ${SITEMAP_ROUTES.length} routes to ${target}`);
}
```

The `/leaderboard` URL gets a **302 (Found, temporary)** redirect to `/`, preserving query string. SvelteKit's `+server.ts` GET handler returns a `Response` with `302` and `Location` header. Sunset 30 days post-cutover (2026-05-30) — the file is deleted then; the test guard from `redirect-sunset.test.ts` fails 14 days BEFORE sunset (2026-05-16) to force operator attention.

> **Why 302, not 301 (architect I4 finding):** The redirect is explicitly TEMPORARY (30-day window). 301 ("permanent") tells crawlers to update their index AND tells browsers it's safe to cache forever; combined with `Cache-Control: public, max-age=86400, immutable` (Plan v1) the browser cache would still issue 301s long after the route file is deleted. 302 says "keep checking back" — exactly the semantics we want. We accept that crawlers MAY take longer to drop `/leaderboard` from their index; that's fine, since the URL still works (via the redirect) for the entire sunset window. Drop `immutable` from cache-control; cap max-age at 1h (3600s) so post-deletion-day clients re-fetch within an hour.

 * 30-day sunset 302 redirect from the pre-cutover URL `/leaderboard`
 * to the post-cutover homepage `/`. Preserves query string verbatim
 * so bookmarked filtered views (e.g. `?tier=verified&sort=...`) land

 * `tests/build/redirect-sunset.test.ts` guard fails CI 14 days BEFORE
 * sunset (2026-05-16) to force operator attention.

 * - 302 (Found, temporary): crawlers keep checking back; browsers cache
 *   only as long as `cache-control` allows. Right semantics for a 30-day
 *   sunset: post-deletion-day, cached clients re-fetch within max-age and
 *   discover the 404.

 *   cache aggressively (combined with `Cache-Control: ...immutable`,
 *   indefinitely). Wrong for a route we're about to delete.
 * - 307/308: preserve method (POST stays POST). Not relevant here —
 *   `/leaderboard` had no POST handler.

export const GET: RequestHandler = ({ url }) => {

  // params or `?foo=bar` (with leading `?`) when params present.

      // within the hour and learn the URL is gone. See architect I4.

```svelte
<script lang="ts">
  /**
   * Layout-level JSON-LD structured data injector. Renders WebSite +
   * Organization schemas in <svelte:head>. Mounted from +layout.svelte
   * so every page emits the same canonical metadata.
   *
   * Why two schemas, not one:
   *   - WebSite: Google sitelinks-search-box eligibility (cmd-K → /search?q=…)
   *   - Organization: brand-knowledge-panel signals (logo, name, URL)
   *
   * Why no per-page schema today: spec §5.8 says JSON-LD for SEO. Per-page
   * Article/Dataset/etc. schemas are richer but require deciding the right
   * type per route. Deferred to P6.
   *
   * pageUrl: the FULLY QUALIFIED canonical URL of the current page.
   * Computed in +layout.svelte from $page.url.href. Passed as a prop so
   * the test can inject deterministic values.
   */
  import { SITE_ROOT } from '$lib/shared/site';

  let { pageUrl }: { pageUrl: string } = $props();

  // JSON-LD XSS hardening helper. Even though today's schemas are static,
  // emitting `</` literally inside a <script> body can break out of the
  // tag. Standard Next.js / Remix idiom is to JSON.stringify then replace
  // `<` with its Unicode escape — the JSON parser still accepts the
  // escape, but no `</script>` sequence can appear in the output. Future
  // schemas with dynamic data (run notes, model descriptions) inherit the
  // safeguard for free.
  function jsonLd(schema: unknown): string {
    const safeJson = JSON.stringify(schema).replace(/</g, '\\u003c');
    return `<script type="application/ld+json">${safeJson}</script>`;
  }

  // Build WebSite schema. The `url` property is the SITE root (NOT the
  // current page) per https://schema.org/WebSite — the WebSite schema
  // describes the whole site; per-page canonical is the <link>.

  const websiteSchema = {
    '@context': 'https://schema.org',
    '@type': 'WebSite',
    name: 'CentralGauge',
    description: 'Open-source benchmark for evaluating LLMs on AL code generation, debugging, and refactoring for Microsoft Dynamics 365 Business Central.',
    url: SITE_ROOT,
    potentialAction: {
      '@type': 'SearchAction',
      target: {
        '@type': 'EntryPoint',
        urlTemplate: `${SITE_ROOT}/search?q={search_term_string}`,
      },
      'query-input': 'required name=search_term_string',
    },
  };

  const organizationSchema = {
    '@context': 'https://schema.org',
    '@type': 'Organization',
    name: 'CentralGauge',
    url: SITE_ROOT,
    sameAs: [
      'https://github.com/SShadowS/CentralGauge',
    ],
  };
</script>

{@html jsonLd(websiteSchema)}
{@html jsonLd(organizationSchema)}
```

> **Why `{@html}`:** Svelte 5 renders `<script>` tags as text by default (XSS protection). Structured data is content-safe (we control the JSON), so `{@html}` is correct here. The `jsonLd()` helper escapes `<` to `<` so the output never contains a `</script>` sequence — defends against future schemas that ingest user-submitted content (model descriptions, run notes) without requiring per-call review. Pattern lifted from Next.js's `<Script>` and Remix's `json-ld` helper.

### Task C3: Add `<link rel="canonical">` per page in layout

**Files:**
- Modify: `site/src/routes/+layout.svelte`

A single `<link rel="canonical">` in the layout `<svelte:head>` populated from `$page.url.href` covers every route. Per Google: canonical should match the URL the page would be shared as. With query params, the canonical typically excludes them (e.g. `/leaderboard?tier=verified` canonicalizes to `/leaderboard`) — but this is a hint, not an absolute. We choose to include the full URL (with query) for filtered views since they ARE distinct content from the unfiltered page (different table rows, different sort).

> **Trade-off considered: include vs strip query params from canonical.**
>
> - **Include**: every URL is its own canonical. Filtered views get indexed separately. Risk: explosion of indexed URLs (the leaderboard's filter combinatorics × ~9 surfaces × ~50 query combos = ~450 URLs).
> - **Strip**: filtered views canonicalize to the unfiltered base. Search engines index one URL per surface. Risk: filter-bookmarkers searching for their filter find the unfiltered page, then have to reapply.
>
> Pick **strip** for P5.5: simpler, fewer indexed URLs, cleaner sitemap relationship. Filter URLs are direct-share linkable but not search-discoverable. If a P6+ design wants filtered views findable, switch to include via a `+layout.server.ts` data field that the canonical reads.
